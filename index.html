ℹ️ Key Terms

IDT (Input Device Transform): Converts various input formats (camera footage, textures, etc.) into our working ACEScg color space. Net Effect: Ensures all source materials speak the same color language, preventing color mismatches between departments.

ODT (Output Device Transform): Converts ACEScg imagery to specific display formats (monitor, projector, TV). Net Effect: What you see accurately represents how it will appear on the target medium, without permanently altering the source data.

ACEScg: Our linear working color space with AP1 primaries. Why It Matters: Wider gamut than sRGB, mathematically predictable for CG lighting calculations, prevents color clipping.

ACES2065-1: Archive format with AP0 primaries. Why It Matters: Future-proof storage that encompasses all visible colors, ensuring remasters are possible decades later.

✨ Future Flexibility — Key Benefit

Since all CG renders and composites are maintained in ACEScg linear color space, any future delivery requirements can be met without re-rendering the entire film.

Example Scenario: Client requests HDR version after theatrical release? Convert the archived ACEScg/ACES2065-1 masters → Rec.2020 PQ, perform trim passes and QC in the DI, then re-export — no shot re-rendering required.

2Production-Critical Implementation Rules

1 Always Work Scene-Referred Author, render, and composite in ACEScg (linear AP1)
Example (Arnold): Shaders in Arnold use ACEScg albedo values, area lights emit in ACEScg linear values, and all textures are converted via sRGB-Texture IDT into ACEScg. Mathematical operations remain linear and physically accurate.
Why This Matters: Linear color space ensures accurate light bounces, proper motion blur calculations, and correct depth of field. Working in display-referred (sRGB) would break physical simulations and create unpredictable results.
2 View Transform, Never Bake Apply ACES ODT only in viewer until final delivery
Example (Nuke): Read EXR (ACEScg) → Set project OCIO to aces_1.3 config → Viewer LUT = ACES 1.0 SDR-video (Rec.709). The EXR file on disk stays ACEScg linear; Rec.709 exists only in the viewer for monitoring.
Example (RV): Load ACEScg EXR sequence → OCIO config.ocio active → Display transform set to P3-D65. The underlying EXR files remain untouched ACEScg linear data.
Why This Matters: Baking viewing transforms = permanent data loss. Keeping everything in ACEScg means infinite re-delivery options without quality degradation.
3 Deliver by Display Medium Each display technology needs specific encoding
Delivery Specifications: • Cinema: DCI-P3 X'Y'Z', gamma 2.6
• HDR TV: Rec.2020 PQ, 1000–4000 nits peak
• Broadcast TV: Rec.709, gamma 2.4, 100 nits
• Web/Mobile: sRGB, gamma 2.2, varied peak
When Baked: Only at delivery stage via Resolve/OCIO export. All upstream work remains in ACEScg.
4 Test Before Production Lock Validate OCIO 2.4.x across all departments
Testing Protocol: Phase 1: Install OCIO 2.4.x on test workstations
Phase 2: Render test asset through full pipeline
Phase 3: Validate color consistency across DCCs
Phase 4: Lock configuration and distribute to all teams
Critical Validation: Version mismatch = color shifts. Test especially: Maya↔Nuke handoff, Substance→Arnold texture pipeline, RV↔Resolve viewing match.
5 Practical Contingencies Real-world fallbacks that actually work
Emergency Protocols: • Pre-baked Rec.709 LUT (.cube) for dailies viewing
• ACEScg→sRGB 3×3 matrix documented and tested
• Resolve project with manual pipeline ready
• Regular drill of emergency workflow
Reality Check: Maintain simple viewing LUTs that can be loaded in any software. Keep masters in ACEScg — viewing can always be solved, but lost data cannot be recovered.
6 Archive = Future Revenue ACES2065-1 protects long-term value
Complete Archive Package: • All frames in ACES2065-1 AP0 (16-bit half EXR)
• Show look LUTs and creative decisions
• OCIO 2.4.x config (frozen copy)
• MacBeth chart reference renders
• Grade metadata and trim pass notes
Business Impact: New display tech or delivery format later? Retrieve from ACES2065-1 and create new trims/ODTs without re-rendering.

3Monitor Hardware Specifications & Configuration

Artist Workstation Monitors

  • Type: sRGB/Rec.709 minimum
  • Coverage: 100% sRGB
  • Bit Depth: True 10-bit panel (not 8-bit+FRC)
  • Calibration: Monthly, ΔE < 3.0
  • Examples: ASUS ProArt PA279, BenQ SW270C

Color Critical / Review Monitors

  • Type: P3-D65 capable reference
  • Coverage: 98%+ DCI-P3
  • Bit Depth: True 10-bit or 12-bit
  • Calibration: Weekly, ΔE < 2.0
  • Examples: EIZO CG319X, FSI DM250

HDR Grading Monitors

  • Type: HDR1000 minimum
  • Coverage: 90%+ Rec.2020
  • Peak: 1000+ nits sustained
  • Calibration: Before each grading session
  • Examples: Sony BVM-HX310, Dolby PRM-4220
NVIDIA GPU 10-BIT CONFIGURATION (FULL-PATH)
NVIDIA Control Panel → Display → Change Resolution
Apply the following settings:
  Output color depth: 10 bpc
  Output color format: RGB
  Output dynamic range: Full

Why: Enables full 10-bit pipeline from GPU to display
HDR SIGNAL PATH VERIFICATION (FULL-PATH)
Windows: Settings → Display → HDR: ON
macOS: Displays → HDR capable display detected
Resolve: Project Settings → HDR10+/Dolby Vision as applicable

Verify: 10-bit 4:4:4 @ 60Hz signal path active

⚠️ Critical: Why 10-bit Matters

8-bit = 256 levels per channel = visible banding in gradients
10-bit = 1024 levels per channel = smooth gradients essential for animation
Without 10-bit enabled, you're seeing less than 25% of the color information in your renders.

5Texture Pipeline Specifications

Albedo/Base Color sRGB PNG/TIF/TGA Utility - sRGB - Texture ACEScg Purpose: Linearizes color for accurate light interaction
Validate: No values > 1.0 after IDT application

Validate: Blue channel ~0.5–1.0, no color transform applied

Validate: Values strictly 0–1, no color management

Validate: Can exceed 1.0 for bloom/glow effects

Validate: Check dynamic range, no negative values
Displacement EXR (32-bit float) Raw (no transform) Data Purpose: Geometric surface detail
Validate: Centered at 0.5 or 0.0 per pipeline specs

7Pre-Production Testing Checklist

Phase 1: Configuration Testing

  • Install OCIO 2.4.x on representative workstations from each department
  • Verify aces_1.3 config loads correctly in all DCCs
  • Test texture IDT workflow: sRGB source → ACEScg conversion → render
  • Confirm GPU 10-bit output enabled on all color-critical workstations
  • Document any application-specific issues or version conflicts

Phase 2: Pipeline Integration

  • Complete test asset: Model → Texture → Lookdev → Light → Render → Comp → Review
  • Verify color match: Maya viewport = Nuke viewer = RV playback = Resolve output
  • Test emergency fallback: Generate and validate .cube LUT for dailies
  • Render MacBeth ColorChecker through full pipeline for validation
  • Test archive and retrieval of ACES2065-1 masters

Phase 3: Lock and Deploy

  • Freeze OCIO 2.4.x version — no updates during production without written approval
  • Deploy config to all machines via IT management system
  • Create reference chart renders for ongoing QC (store with assets)
  • Department heads sign-off on color pipeline before first shot
  • Schedule regular validation meetings for first quarter of production

8Stylized Animation Considerations

📎 CGI Animation vs VFX Pipeline Differences

Key Distinction: CGI stylized animation doesn't use CDL (Color Decision List) workflows from live-action VFX.

Instead of CDLs, we use:

  • Look Development LUTs: Artistic style choices created during pre-production
  • Show Look Transforms: Applied as non-destructive view layers
  • Grade Nodes: Shot-specific adjustments in Nuke or Resolve
  • Creative Metadata: Document artistic intent separately from technical pipeline

Why Different: Live-action VFX uses ASC-CDL to match photographed plates. Pure CG animation creates its look from scratch, requiring different creative tools while maintaining the same technical color pipeline.
